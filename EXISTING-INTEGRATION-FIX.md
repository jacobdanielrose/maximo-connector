# Fix for Existing Integration with Auth Errors

## Your Situation

- ✓ Integration created through UI
- ✓ Connector bridge is running
- ✗ Getting `UNAUTHENTICATED: invalid client_id or client_secret` errors

This means the gRPC credentials between your connector and the bridge are out of sync.

## Root Cause

The `connector` secret exists but contains invalid or stale credentials. This typically happens when:
1. The connector bridge was restarted/updated after the integration was created
2. The integration was created before the bundle was fully deployed
3. There's a mismatch between the connector instance ID and the bridge's expected credentials

## Solution: Force Credential Regeneration

### Option 1: Delete and Recreate Integration (Recommended)

This is the cleanest approach:

```bash
# Set your namespace
NAMESPACE=cp4waiops  # or your actual namespace

# 1. Find your connector configuration
oc get connectorconfiguration -n $NAMESPACE

# 2. Delete it (replace <config-name> with actual name, e.g., maximo-prod)
oc delete connectorconfiguration <config-name> -n $NAMESPACE

# 3. Wait for the secret to be cleaned up (should take 5-10 seconds)
oc get secret connector -n $NAMESPACE
# Should return: Error from server (NotFound)

# 4. Restart the connector pod to clear any cached state
oc delete pod -n $NAMESPACE -l app=ticket-template

# 5. Now recreate the integration through the UI
# Go to AIOps UI → Integrations → Add Integration → IBM Maximo
# Use the SAME configuration as before
```

### Option 2: Force Secret Regeneration (If Option 1 Doesn't Work)

If deleting the configuration doesn't remove the secret:

```bash
NAMESPACE=cp4waiops

# 1. Delete the connector secret manually
oc delete secret connector -n $NAMESPACE

# 2. Delete the connector configuration
oc delete connectorconfiguration <config-name> -n $NAMESPACE

# 3. Restart connector pod
oc delete pod -n $NAMESPACE -l app=ticket-template

# 4. Wait 30 seconds for cleanup

# 5. Recreate integration through UI
```

### Option 3: Restart Connector Bridge (Last Resort)

If the above don't work, the bridge itself may need to be restarted:

```bash
NAMESPACE=cp4waiops

# 1. Restart the connector bridge
oc delete pod -n $NAMESPACE -l app.kubernetes.io/name=connector-bridge

# 2. Wait for bridge to come back up (check with: oc get pods -n $NAMESPACE | grep connector-bridge)

# 3. Delete your integration
oc delete connectorconfiguration <config-name> -n $NAMESPACE

# 4. Delete connector secret
oc delete secret connector -n $NAMESPACE

# 5. Restart connector pod
oc delete pod -n $NAMESPACE -l app=ticket-template

# 6. Recreate integration through UI
```

## Verification Steps

After recreating the integration:

### 1. Check Secret Was Created
```bash
oc get secret connector -n $NAMESPACE -o yaml
```

Should show keys: `id`, `client-id`, `client-secret`

### 2. Watch Connector Logs
```bash
oc logs -n $NAMESPACE -l app=ticket-template -f
```

Look for:
- ✓ `starting configuration consume stream: channel=grpc-connector-configuration-updates-channel`
- ✓ `CloudEventProducer created`
- ✓ NO `UNAUTHENTICATED` errors

### 3. Check Configuration Status
```bash
oc get connectorconfiguration <config-name> -n $NAMESPACE -o yaml
```

Look for:
```yaml
status:
  conditions:
  - type: Ready
    status: "True"
```

## Why This Happens

The gRPC authentication uses a client ID and secret that are:
1. Generated by the connector bridge when you create an integration
2. Stored in the `connector` secret
3. Used by your connector pod to authenticate

If any of these get out of sync (e.g., bridge restarts, secret corruption, timing issues), you get authentication failures.

## Prevention

To avoid this in the future:
1. Always deploy the bundle manifest BEFORE creating integrations
2. Don't manually edit the `connector` secret
3. If you need to restart the bridge, recreate all integrations afterward
4. Use unique names for each integration to avoid conflicts

## Still Having Issues?

If you're still seeing authentication errors after trying all options:

1. **Check for multiple connector instances:**
   ```bash
   oc get connectorconfiguration -n $NAMESPACE
   ```
   Delete any duplicates or old instances.

2. **Verify bundle manifest is deployed:**
   ```bash
   oc get bundlemanifest maximo-connector
   ```
   Should show: `Configured`

3. **Check connector schema:**
   ```bash
   oc get connectorschema maximo-connector -o yaml
   ```
   Verify it exists and has no errors.

4. **Review bridge logs for clues:**
   ```bash
   oc logs -n $NAMESPACE -l app.kubernetes.io/name=connector-bridge --tail=100
   ```

5. **Check for admission webhook errors:**
   ```bash
   oc get events -n $NAMESPACE --sort-by='.lastTimestamp' | tail -20
   ```

## Expected Timeline

- Secret deletion: Immediate
- Pod restart: 30-60 seconds
- Integration recreation through UI: 2-3 minutes
- Authentication success: Within 10 seconds of pod starting

## Success Indicators

You'll know it's fixed when:
1. No `UNAUTHENTICATED` errors in connector logs
2. You see `starting configuration consume stream` messages
3. The connector starts polling Maximo (if in live mode)
4. Integration status shows "Connected" in the UI
